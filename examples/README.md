### Quality progression round-15 → round-16

The R16-A fix produced a **66% increase in average note count**
(64 → 106) at the same epoch count and same temperature. This is
because the round-15 "val=1.70" was actually the resumed second
`trainer.train()` call from val=1.70 with fresh warmup — those
extra epochs were near-no-op. With the bug fixed, every epoch now
actually trains on the current schedule.

### Note count vs loss progression

Round-17 dropped val_loss from 1.65 (round-16) to 1.10, the largest
single-round improvement so far. Note count actually went DOWN at
the same temperature (106 → 54) because lower loss means the model
terminates sequences earlier (more confident about when to end). This
is the expected behavior: better-trained models produce more focused,
shorter outputs at a given temperature, not longer ones.

## How to regenerate any example

```bash
python -m samar.generating \
  --latent-index 65 \
  --max-length 256 \
  --temperature 1.0 \
  --output-xml examples/11_huzam_hafez_t10.xml \
  --output-events examples/11_huzam_hafez_t10.txt
```

## Known limitations (current model)

- **Empty measures**: Some files have 20-30% empty measures (model emits
  notes without Bar tokens between them). Round-10 reconstructor splits
  these into separate measures but they remain empty. This is a data-
  side issue (model needs more training to learn Bar placement).
- **Velocity_16 dominates**: Hardcoded velocity=64 in parser (round-2).
  Model overfits to Velocity_16. Future: parse `<velocity>` from XML.
- **No dynamic velocity variation**: All notes are Velocity_16.
- **Max ~114 measures**: Limited by 256-token generation window.
