## How to regenerate any example

```bash
# Example: Huzam Hafez with temp=1.0, top_k=40
python -m samar.generating \
  --description-source data/xml/abdel_halim_hafez_qululu_alhaqiqa_huzam_1958.xml \
  --latent-index 12 \
  --temperature 1.0 \
  --top-k 40 \
  --output-xml examples/05_huzam_hafez_t10.xml \
  --output-events examples/05_huzam_hafez_t10.txt
```

## Quality observations

Across all 6 examples (see per-example notes below):

- **All 6 produce 256 tokens** (full max_length, no early <pad>).
- **Token diversity varies**: 19 (deterministic) to 43 (random).
- **Microtones**: Only sharps (`<alter>1.0</alter>`). **No half-flats
  in this batch** -- the model didn't happen to pick the quarter-tone
  E half-flat that we saw in the earlier /tmp/gen5.xml test.
- **Instruments**: Voice, Piano, Violin appear in all 6. Harp
  appears in 1. **Drumset appears in 0** in this regenerated batch.
- **Velocity bias**: All examples dominated by `Velocity_16`. This
  is the round-2 known issue: velocity=64 was hardcoded in the
  parser. Real data probably has more variety; the model learned
  the bias.
- **Bars**: All measures now numbered `1` (MuseScore-friendly)
  with `<attributes>` block (time sig, key, clef) added. Earlier
  regeneration had `measure number="0"` which MuseScore renders
  as empty -- the round-7 reconstructor fix renumbers 0->1.

## Known limitations (round 6 model)

- **Velocity_16 dominates** (round-2 hardcoded velocity=64 in
  parser). To fix: parse `<velocity>` from XML or compute from
  event stream.
- **No `Bar_N` advance** within 256-token window. Generation starts
  with `Bar_0` and stays there. Real pieces need ~100 bars. With
  context_size=256 and most events being notes/durations, the model
  doesn't reach Bar_1, Bar_2 etc.
- **Only `Alter=1.0` and `Alter=-0.5` and `Alter=0.5` are in the
  vocab** (from FIGARO's 24-EDO alter=+/-1.0 plus the round-2
  +/-0.5 quarter-tone extensions). Model emits +/-1.0 freely but
  rarely +/-0.5 with the round-6 checkpoint.
